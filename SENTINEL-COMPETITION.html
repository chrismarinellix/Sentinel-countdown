Anonymous Contributor ‚Ä¢ Systems & Scripting ‚Ä¢ Submitted Nov 14, 2025

Not Competing

üì¢ Note: Submitted for improvement purposes only - not competing for prize. Submitter wishes to give mid-level and senior engineers a chance to compete and boost their interest level.

üìã Current Situation: Every project has two stages - one is running script with PSCAD which ends up in 100GBs of data on big machines. When a task is done in big machine, data transferred to SSD (which is faster), then access PSSE straight from SSD without copying - this would boost the work significantly.

‚è±Ô∏è Time Saved

6-9 hours per test cycle

üíµ Investment

SSD hardware + network setup

‚ö° Impact

All projects benefit immediately

üéØ Strategy

Manage individual SSDs per machine

üéØ Why This Matters: Eliminates waiting time for copying data between machines (6-9 hours saved per cycle). High productivity boost across all projects. No waiting for data transfer - engineers can access PSSE data directly from network drive. Smart strategy with upfront investment cost but massive time savings across the team.

üìä Entry #4: Standardized Simulation Assessment Guideline

Dominic Moncada ‚Ä¢ Power Systems Engineer ‚Ä¢ Australia ‚Ä¢ Quality & Standards ‚Ä¢ Submitted Nov 14, 2025

Under Review

üìã Current Situation: After we run simulations, we submit to a senior/lead/principal for internal review, but each reviewer reviews work differently. Delivered work is not consistent in quality - depends on scrutiny of the reviewer. Review time is not consistent - some reviewers may require more or less time to review the same type of simulation, depending on their expertise and methodology. Form a Technical Working Group (TWG) that will meet on a regular basis (weekly/fortnightly) to develop the document until it is completed and rolled out across the team.

Reduced review hours per project

üíµ Cost Reduction

Lower charge-out rate via peer reviews

‚ö° Quality Impact

Consistent work outputs

üéØ Risk Reduction

Less rework, fewer errors

üéØ Why This Matters: Consistent work outputs because everyone reviews the same way - all delivered work will be of the same quality. Consistent review time because the review method is established - we can estimate review duration and confidently communicate to clients. Documented guideline reduces dependence on people and shifts focus to process - maintains business continuity if staff leave. Peer reviews between engineers can happen with confidence, significantly reducing workload of senior/lead/principal engineers. Builds technical capability among all engineers, enabling them to reflect on results and make valuable engineering judgements.

üìã Current Situation: In initialization tests, models must settle to steady-state values within a specified time frame (e.g., 3 seconds). Currently, engineers manually check each test case by reviewing plots of reactive power Q at the point of connection (POC Q) to verify compliance. This manual review process is time-consuming, especially for projects with many test cases.

‚úÖ Proposed Solution: Modify the initialization script to automatically flag any test case that does not initialize within the specified time frame. Instead of engineers manually checking each test case by reviewing plots, the script can expedite the process by programmatically detecting non-compliant cases. This is one example of how we can standardize and speed up the review/assessment process for simulations.

‚è±Ô∏è Time Saved

üéØ Accuracy

Consistent compliance checks

‚ö° Scalability

Handles any number of test cases

Entry #4: Assessment Guidelines

üéØ Why This Matters: This automated approach transforms manual, time-consuming compliance checks into instant, programmatic assessments. Engineers can focus on analyzing results rather than verifying compliance. The script-based flagging ensures no non-compliant test cases slip through, improving quality and consistency while significantly reducing review time.

üìä Example: A plot of POC Q (MVAr) vs time shows a model settling to 0 MVAr after ~4 seconds. If the guideline specifies 3 seconds, the script automatically flags this as non-compliant - no manual inspection required.
